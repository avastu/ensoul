# Visual Primitives

Use this reference when an artifact needs a grammar: a reusable language of marks, spaces, thresholds, and compositions. The goal is not to imitate any source image. The goal is to extract portable visual operations that let an agent build ARTifacts with presence.

## Primitive Doctrine

A primitive is a mark with a job. It is smaller than a template and more durable than a style.

Good primitives:
- encode a relationship, not a mood;
- can be recombined;
- make the artifact easier to remember;
- can survive ASCII, SVG, Mermaid, tables, or image prompts;
- carry truth before decoration.

## Temple-Derived Grammar

Inspired by Works in Progress' "Modern Hindu temples" article and images: nested marble ceiling geometry, axial temple approaches, spires, plinths, carved narrative bands, load-bearing stone, numbered logistics, volunteer/artisan collaboration, and the article's central inversion that a temple is a residence for presence rather than an auditorium for an audience.

Do not copy sacred forms casually. Translate structural principles respectfully into abstract visual grammar.

### 6. Carved Frame / Patterned Border

Use when the artifact needs a visual world before the viewer reads the center. A rich border can encode context, pressure, evidence, repetition, lineage, or constraints. It should behave like temple carving or illuminated marginalia: the edge teaches the center.

Default to **wordless pattern**. Use labels inside a frame only when the frame's meaning would otherwise be ambiguous or when the artifact is explicitly instructional. Most frames should be felt before they are decoded.

Bad frame:

```text

# decorative box only  #
########################
```


Density controls:
- **Light frame:** sparse repeated marks with lots of air; best for presence, grief, awe, or elegance.
- **Medium frame:** a consistent motif around all edges; best default.
- **Heavy frame:** dense carved border; use only when the edge is part of the argument.

Frame rules:
- The top border can name aspiration, question, or atmosphere.
- The side borders can carry constraints, guardians, or repeated pressures.
- The bottom border should often carry the plinth: cost, evidence, labor, debt, or provenance.
- Avoid using more than two border vocabularies unless the collision is the point.
- A frame can be asymmetrical when the subject is unstable.
- Prefer wordless motifs over border words. If words appear in the frame, keep them rare and load-bearing.

## Coding-Collaboration Grammar

Use this when turning a coding session, bug hunt, implementation plan, PR review, deployment issue, or transcript into a compact state artifact. The goal is to let the human recognize the state of the work faster than prose.

Coding marks should be instinctive before they are explained. Prefer marks with visual affordance:

- a check should mean verified;
- a document-like block should mean file, log, or receipt;
- a broken cross should mean failure;
- a delta should mean change;
- a question mark should mean unknown;
- a wave should mean unstable or still moving.

### Coding Primitives

| Mark | Meaning | Use when... |
| --- | --- | --- |
| `вң“` | verified proof | tests passed, command succeeded, behavior observed |
| `в–Ө` | file / receipt / evidence | naming files, logs, screenshots, artifacts, source links |
| `вң•` | failure / regression / risk | showing broken behavior, bugs, bad assumptions |
| `!` | urgency / broken path | production issues, deadlines, blocked default path |
| `О”` | change / patch | what was modified, transformed, or moved |
| `в—Ҷ` | design / decision / stable pattern | architecture, conventions, intended shape |
| `oв”Җв”Җo` | dependency / workspace / handoff | showing agents, services, repos, owners, worktrees |
| `?` | unknown | unanswered question, missing evidence, unresolved cause |
| `~` | unstable / investigation weather | flaky behavior, moving state, partial confidence |
| `в–І` | user-visible feature | visible behavior rising from foundations |
| `в—Һ` | behavior change at center | the actual requested behavior or semantic core |
| `в–Ғв–Ӯв–ғ` | implementation layers | repo pattern, state/data, UI, tests, release |

### Transcript Compression

When summarizing a past coding transcript:

1. Identify the center `в—Һ`: the actual behavior or decision the work changed.
2. Put receipts `в–Ө` at the edge: files, logs, commands, links, screenshots.
3. Put verified proof `вң“` only where something was observed or tested.
4. Put failures `вң•` where they occurred: false starts, regressions, bad assumptions.
5. Put unknowns `?` or instability `~` only where unresolved.
6. Use one canonical frame, not a collage of all frames.
7. End with one state label: `not real-env verified`, `implementation-tested`, `ready for review`, `branch pushed`, `merged`, or `recognition-pending`.
8. Never mark a thing verified because it exists in the grammar. Verification means an observed run, test, screenshot, command result, review, or human recognition. If only one sample was tested, say `one sample tested`.
9. When naming untested/open frame families, use exact names from this reference. Do not promote loose primitives like `orbit` or `vessel` into tested frame families unless a frame with that name exists.

The artifact should help the human answer: "Where are we, what changed, what was proven, and what remains?"

## Taste Rules

- One dominant primitive per artifact. Supporting primitives should orbit it.
- If using sacred or culturally specific inspiration, abstract the operation and avoid pretending to reproduce the tradition.
- Let ornament carry evidence, sequence, or boundary.
- Use symmetry for reverence, stability, ceremony, or inevitability.
- Break symmetry only to show rupture, life, danger, or choice.
- Put mundane constraints at the base or edge; put living presence at the center or aperture.
- Do not over-label. A primitive should teach the eye.

## Rigor Tests for Primitive-Based Artifacts

Apply these before shipping any visual artifact, especially weird ASCII, symbolic mirrors, coding-state cards, and temple diagrams.

### Eraser Test

For each mark, ask whether removing it would lose one of:
- information;
- comparison;
- evidence;
- sequence;
- boundary;
- felt structure;
- useful rhythm.

If removing the mark changes only decoration, remove it.

### Collision Test

For each text label, leader line, border motif, dense glyph cluster, and annotation:
- imagine its bounding box;
- check whether another mark crosses or crowds it;
- check whether the frame competes with the center;
- check whether labels are forced into dense data or dense ASCII texture.

Standard repairs:
- move prose below the artifact as a caption or evidence drawer;
- reduce frame density from heavy to medium or light;
- separate labels from marks with leader lines or whitespace;
- move repeated labels into a strip or legend;
- use direct labels instead of a legend when it reduces eye travel;
- remove duplicate encodings.

### Integrity Test

Visual emphasis must match evidence and importance.
- Do not put unverified claims at the center.
- Do not mark something `вң“` unless it was actually observed or tested.
- Do not invent alternate verification symbols like `в—‘ pending` unless the artifact defines them in a legend. Prefer existing marks: `~` for unstable, `?` for unknown, `!` for urgency.
- Do not make uncertainty look like proof.
- Do not let a beautiful frame imply completeness.
- If a comparison matters, show the baseline, alternative, before/after, or counterfactual.
